## vc49-QA sweep — Ship 1 of 3 (HIGH data-integrity trio) implemented, NOT yet shipped

Full-app code QA at vc49 (`docs/qa-code-review-2026-07-03.md`, Fable 5) produced 9 briefs. Grouped
into 3 ships by risk. **Ship 1 = the three HIGH-severity data-integrity fixes**, verified vs live
source first, built together (`assembleDebug` **SUCCESSFUL**, 10m37s). All → `docs/DONE/`. Not yet
version-bumped/dispatched (still vc49 on Play).

- **drive-ids-alignment** (QA H1) — `NewEditEntryScreen.kt`, `EntryDetailScreen.kt`. Keeps the
  positional `driveFileIds` list aligned with `photoUris` through every photo edit. Added a parallel
  `driveIds` rememberSaveable state (padded to photoUris length with `""`), mirrored through
  picker-add, camera-add, remove, drag-reorder, and passed as `driveFileIds` on Save (was
  `existingEntry?.driveFileIds` = original order). Detail screen: new `paddedDriveIds()` helper;
  `withCover`/`withRemoved`/`withSwapped` now pad-then-mutate unconditionally (previously skipped the
  driveIds mutation when the list was shorter, desyncing pending-upload photos).
  *Deviation:* preserved the `sessionAdded` file-cleanup block in the remove handler that the brief's
  BEFORE snippet omitted; dropped `.distinct()` on adds per the brief (persistToGallery always makes a
  unique copy, and distinct() would break the parallel-list invariant).
- **save-entry-upload-race** (QA H2) — `JournalViewModel.kt`. Background Drive-upload completion no
  longer re-saves the stale captured entry (which reverted mid-upload edits). `saveEntry` and
  `syncPhotosToGoogleDrive` now merge only `driveFileIds` into the *live* entry looked up by id, and
  skip the write if `photoUris` changed since upload start (indices no longer match; next save
  re-uploads). No deviations.
- **photo-exif-rotation** (QA H3) — `ImageStorage.kt`, `JournalBackup.kt`, `libs.versions.toml`,
  `app/build.gradle.kts`. Camera photos no longer render sideways: re-encode paths now read the EXIF
  Orientation tag and bake the rotation into the bitmap before compressing (BitmapFactory ignores the
  tag and Bitmap.compress writes none). Added `androidx.exifinterface:1.3.7`; new
  `ImageStorage.exifOrientation` (private) + `applyExifOrientation` (`internal` so `JournalBackup`
  reuses it) wired into `compressForStorage` and compact-export `compressToBytes`. No deviations.
