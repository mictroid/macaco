## On-device signed-release QA pass ‚Äî found and fixed a real R8 gap (`app/proguard-rules.pro`)

Built the signed release APK (`assembleRelease`) and installed on the S8+ (`ce03171353d9547e0c`) for
the `enable-r8` brief's required QA pass. **App crashed on launch, before `MainActivity` ever rendered**
‚Äî confirms the brief's own warning that "R8 bugs are runtime-only." `adb logcat` showed:

```
FATAL EXCEPTION: main
java.lang.RuntimeException: Unable to get provider androidx.startup.InitializationProvider
Caused by: java.lang.NoSuchMethodException: androidx.work.impl.WorkDatabase_Impl.<init> []
```

**Deviation from the brief:** the brief's keep rules covered the Google Drive API client and
kotlinx.serialization as reflection risks but missed **WorkManager** (used by `ReminderScheduler` for
periodic "write an entry" reminders), which persists scheduled work via an internal Room database.
Room instantiates its generated `*_Impl` class via `Class.getDeclaredConstructor()` reflection ‚Äî R8 was
stripping `WorkDatabase_Impl`'s no-arg constructor. Added, not in the original brief:

```proguard
-keep class * extends androidx.room.RoomDatabase
-keep class androidx.work.impl.** { *; }
-dontwarn androidx.work.**
```

Rebuilt, reinstalled, relaunched ‚Äî no crash, app opens cleanly. **Still need to complete the rest of the
brief's 6-point QA checklist** (Drive sync, local backup export/import, RevenueCat paywall/purchase/
restore, maps, Crashlytics deobfuscation) before shipping ‚Äî only the launch-crash was caught and fixed
so far.

Getting this one out took **4 dispatch attempts** across two distinct, unrelated infra bugs ‚Äî neither
was in the app code:

1. **Run `29312081942`** (08:40:43 +0200) ‚Äî FAILED. `invalid_grant: ID Token issued at ... is stale
   to sign-in` at the `publishReleaseBundle` upload step. Root cause: `.github/workflows/release.yml`
   fetched the short-lived GitHub OIDC token *before* the Gradle build; R8/minification (this same
   batch) pushed `bundleRelease` to ~6-7 min, and the token had expired by the time the upload step
   tried to use it for the Google STS exchange. Worked fine pre-R8 because builds were fast enough to
   stay inside the token's freshness window.
2. **Run `29312725271`** (08:53:06 +0200) ‚Äî retried with no changes, **same failure** (rules out a
   one-off blip; confirms it's structural).
3. **Fix commit `21c812e`** ‚Äî split the workflow into `bundleRelease` (build) ‚Üí fetch OIDC token ‚Üí
   `publishReleaseBundle` (upload only), so the token is minted right before the fast upload instead
   of before the slow build. Pushed, verified `origin == HEAD`.
4. **Run `29313992330`** (09:16:07 +0200) ‚Äî the token-freshness fix worked (got past upload this
   time), but hit a **new, unrelated** 403: `"Release in track targeting no countries"` /
   `PERMISSION_DENIED` at the `commitEdit` step. Root cause: this is the **first-ever release to the
   Open testing track**, and that track had never had country/region availability configured in Play
   Console ‚Äî a one-time chicken-and-egg gate the API enforces on first use, same pattern as the
   media-permissions declaration gotcha (see `play-media-permissions-declaration` memory). Not fixable
   via CI/code ‚Äî required a manual Play Console UI step (Testing ‚Üí Open testing ‚Üí Countries/regions).
   User set this manually.
5. **Run `29325050510`** (10:21:26 UTC / 12:21:26 +0200) ‚Äî ‚úÖ **SUCCESS**, 9m37s. Bump+notes commit
   `b429e3d` (vc70‚Üí71, versionName 1.6 unchanged); WorkManager/Room fix `e0c2f62`; original 2-brief
   batch `f24a7a3`; CI token-freshness fix `21c812e`. **PUBLISHED to Open testing.**

**Takeaways for future releases:** (a) any future build-time regression (more R8 rules, added
dependencies, etc.) risks reintroducing OIDC token staleness if the build ever creeps back toward the
token's freshness window ‚Äî the split-step structure in `release.yml` is the durable fix, don't
recombine build+publish into one Gradle invocation. (b) Any **new Play track** (first use of open
testing, or later production) may need one-time manual Play Console setup (countries, in this case)
before the API can publish there ‚Äî check this proactively before dispatching to a track for the first
time, not reactively after a 403.
