## map-camera-v11 — on-device diagnosis + fix (HELD for next batch, not pushed)

Tested v10 on the A53 via **wireless debug** (`adb mdns services` → `adb connect`; signed in, premium
via RevenueCat promo entitlement; debug build over uninstalled Play build). v10's added Logcat
diagnostics did their job and revealed the **real, long-standing root cause** behind the v1–v10 saga:

- **`cameraPositionState.move()` SILENTLY CLAMPS to the SDK minimum zoom.** Logcat showed v10 computed
  zoom 2.60, but the camera settled at **3.0** — no exception, so every prior "successful" version
  looked wrong. At zoom 3.0 on this xxhdpi device only **~72° of longitude** is visible, so only 1 of
  4 globe-spanning pins (Germany) showed. Our set spans **211.9°**.
- **Two real bugs found, both fixed in v11** (commit `d5638d7`, `MapScreen.kt`):
  1. **Density factor missing.** World width at zoom z = `256·density·2^z` PHYSICAL px; v10 used `256`
     without density, computing zoom ~`log2(density)≈1.4` too high. Fixed via `tile = 256·density`.
  2. **No `minZoomPreference`.** Unset → SDK default floor ~3.0. Set `minZoomPreference = 1f` on
     `MapProperties` → floor drops to ~2.0, visible longitude ~144°, now framing **2 pins**
     (Iceland+Germany) centered.
- **Residual SDK/portrait limit (not fixable with math):** a full-height portrait map won't zoom out
  below ~2.0 (the SDK refuses to show past the poles), capping visible longitude at ~144°. A set wider
  than that (ours is 212°) frames the most pins possible, centered; the 1–2 extreme pins (Argentina,
  Tokyo) sit just off-screen. **User chose "ship the best-effort improvement"** (via AskUserQuestion)
  over off-screen indicators or shrinking the map.

On-device evidence (Logcat `MapCamera`): v10 req 2.60→applied 3.0 (1 pin); +density req 1.21→applied
3.0; +minZoomPreference req 1.21→applied **2.0** (2 pins). v11 final: clean single Log.d line, temp
after-move probe removed, builds green, 2 pins framed and centered, no crash.

**Net:** v11 materially improves the common case — any set spanning ≤~144° now frames ALL pins
perfectly (was capped at ~72°). Truly globe-spanning sets are an accepted SDK limit. Logging kept
(Log.d span/zoom/center + Log.e on move failure) per the brief until the map is confirmed stable in
the wild.
