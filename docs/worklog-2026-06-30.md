### What it adds
A premium 🎬 button on each trip header (`TripHeader`) renders that trip's photos into a 9:16
720×1280 H.264 MP4 (Ken Burns pan/zoom + cross-dissolve) via a new `AdventureReelEncoder`
(MediaCodec input surface → Canvas render → `MediaMuxer` → `cacheDir`), then opens the Android share
sheet with a `FileProvider` URI. New `ReelState` sealed class + `startReel`/`cancelReel`/`reelConsumed`
on the ViewModel, a blocking `ReelProgressDialog`, a `cache-path` entry in `file_paths.xml`, and
`reel_*` strings in all 11 locales.

### Verification (per cowork-brief-workflow) + deviations
Verified vs live source first. The 2 originally-held compile blockers (the brief once used
`getApplication()` on a plain ViewModel, and referenced an undeclared `reelEncoderJob`) were already
fixed in the corrected draft (`appContext` at `JournalViewModel.kt:38`; `reelEncoderJob` declared).
**6 further fixes folded in during implementation:**

1. **`isPurchased` was NOT collected in `JournalListScreen`** — the brief asserted it already was
   (false). Added `val isPurchased by viewModel.isPurchased.collectAsState()`; it's `StateFlow<Boolean?>`
   so the `TripHeader(isPurchased: Boolean)` call passes `isPurchased == true`.
2. **Duplicate `val context`** — Change 6 re-declared `val context = LocalContext.current`, but it
   already exists at `JournalListScreen.kt:122`. Reused the existing one.
3. **Missing imports** — added `android.content.Intent`, `material.icons.filled.Videocam`, M3
   `AlertDialog` / `LinearProgressIndicator` / `TextButton` in JournalListScreen; `kotlinx.coroutines.Job`
   and `android.net.Uri` (and `R`) in JournalViewModel. `ReelState` imported as a nested type.
4. **Crossfade couldn't blend** — the draft's `renderFrame()` locked the surface, `drawColor(BLACK)`,
   drew one layer, and posted, *per call*; the crossfade called it twice (prev, then next), so the two
   layers became two separate posted frames and the second's black clear erased the first. Rewrote to a
   `postFrame { canvas -> ... }` helper that locks once and a `drawKenBurns(canvas, …)` that does NOT
   lock — so the crossfade composites prev+next into a single posted frame and actually blends.
5. **PTS drift** — the draft set `bufferInfo.presentationTimeUs` from the live render counter
   (`frameIndex`), which lags the encoder's output and yields non-monotonic timestamps. Replaced with a
   monotonic `muxedFrames` counter incremented at mux-write time → evenly-spaced, monotonic PTS.
6. **Localized error** — `startReel`'s no-photos error now uses `R.string.reel_no_photos_error`
   (via `appContext.getString`) instead of a hardcoded English string.

Strings inserted with the bulk-localize PowerShell recipe. **Encoding gotcha hit + recorded:**
PowerShell 5.1 treats the typographic quote codepoints U+2018/2019/201C/201D as string delimiters, so
double-quoted `"…“%1$s”…"` literals terminated early and broke the parse. Fixed by using
single-quoted PS strings (literal `$`, so `%1$s` survives), dropping the decorative quotes around the
placeholder, and ASCII-escaping the FR/NL apostrophes as `\'` for aapt. Verified all 11 locales via
Grep (glyphs + `%1$s` intact).

## map-camera-v12 implemented (shipped in vc45)

Cowork brief `code-brief-map-camera-v12.md` implemented and moved to `docs/DONE/`.

- **Commit:** `70882d8` — `feat(map): v12 — surface SDK zoom clamp + globe-spanning hint`
- **Files:** `ui/screens/MapScreen.kt`, `res/values*/strings.xml` ×11 (`map_globe_spanning_hint`).
- **Build:** `assembleDebug` ✅ SUCCESSFUL.

### What it does
v11 confirmed the camera math is correct but the Google Maps SDK clamps `move()` to a portrait
Mercator floor (~zoom 2.0, ≈144° visible longitude) that cannot frame a globe-spanning pin set (the
user's 4-pin set spans ~209°). Rather than fight an unbeatable physics constraint, v12 makes it
**visible + explained**: after `move()` it reads back `cameraPositionState.position.zoom`, logs
`v12: applied zoom=… (requested=…) clamp=YES|no`, sets a new `globeSpanning` flag, and shows a
"Swipe to see all pins" subtitle in the portrait map header when clamped. No camera-math change.

### Verification (per cowork-brief-workflow) + deviation
BEFORE block at lines 234–239 matched live source; `cameraPositioned` decl + the portrait header
"X of Y locations mapped" Text matched. **Deviation:** the brief's Change-1 AFTER code referenced
`zoom` at the `move()` block, but `zoom` is scoped inside the multi-pin `else` branch and is out of
scope there — it would not compile. Fixed by hoisting a `requestedZoom` var (default 6f, set in both
the single-pin and multi-pin branches) and comparing the applied zoom against that. Also added the
missing `androidx.compose.ui.res.stringResource` import. Hint added to the portrait header only
(the landscape header is a single slim row).
