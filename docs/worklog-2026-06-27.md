## photo-compression brief — IMPLEMENTED + COMMITTED local-only (NOT shipped; batching)

Commit `d0467b6`. Verified vs live source first — all 3 BEFORE snippets matched exactly (API 29+
copyTo at 73-75, API≤28 copyTo at 91-93) and none of the 3 imports were present (brief accurate).
`assembleDebug` BUILD SUCCESSFUL. Brief → `docs/DONE/`. Held unshipped per [[batch-before-shipping]]
(1st change in the post-vc40 batch).

**ImageStorage.kt only.** Added private `compressForStorage(bytes, maxDim=1920, quality=85)`: two-pass
decode (bounds → power-of-2 `inSampleSize`, `RGB_565`) then fine-scale to ≤1920px longest side and
re-encode JPEG 85 → a 12MP camera photo (~5-10MB) stored at ~300-600KB (~90% smaller). Drive upload +
backups benefit automatically (smaller stored file). `persistToGallery` raw `copyTo` replaced with
read→compress→write in BOTH API paths; `compressForStorage` returns null for unrecognised formats
(e.g. GIF) → ORIGINAL bytes written as fallback, so no photo is ever lost.

- **Deviation:** guarded the brief's `createScaledBitmap(...).also { sampled.recycle() }` with
  `if (it !== sampled)` — an identity return would otherwise recycle the bitmap we then compress
  (defensive; scaling is guaranteed on that branch).
- **Known follow-up (flagged, out of brief scope):** `persistToGallery` is called on the MAIN thread
  in the picker/camera result lambdas (`EntryDetailScreen:171`, `NewEditEntryScreen:176/189`). The new
  bitmap decode is CPU work; for large multi-selects this can jank. The brief scoped to ImageStorage.kt
  only, so moving the call sites to `Dispatchers.IO` is left as a follow-up (raised with the user).
