## IMPLEMENTED, not shipped — vc72 (1.6) — email verification gate

**code-brief-email-verification.md** (`UserProfile.kt`, `AuthRepository.kt`,
`FirebaseAuthRepository.kt`, `MockAuthRepository.kt`, `JournalViewModel.kt`, `NavGraph.kt`, new
`VerifyEmailScreen.kt`, strings ×11): email/password signups now must click the Firebase
verification link before entering the journal. `UserProfile.emailVerified: Boolean = true` (default
keeps Google accounts, which are pre-verified, and any indifferent call site unaffected);
`createAccount()` calls `sendEmailVerification()` right after account creation;
`toUserProfile()` maps `emailVerified = provider == Google || isEmailVerified`. New
`AuthRepository.sendEmailVerification()` (resend) / `reloadAndCheckEmailVerified()` (refreshes
Firebase state and manually pushes it into `_currentUser`, since `reload()` doesn't fire the auth
listener) — mirrored in `MockAuthRepository` (new email/password mock accounts start unverified;
the second `reloadAndCheckEmailVerified()` call flips them verified, so the flow is testable without
Firebase). `NavGraph`'s gating `when` gained a
`currentUser?.emailVerified == false -> VerifyEmailScreen(...)` branch between the "not logged in"
and "full journal" cases. Verified against live source first — signatures, imports, and the real
(more elaborate, multi-gate) `NavGraph` structure all matched the brief's simplified BEFORE/AFTER
once the elided gates (onboarding/splash/app-lock) were accounted for.

**Deviation (real bug in the brief's own code sample, not a stale-repo drift):** the brief's
`VerifyEmailScreen.kt` snippet referenced `it.getString(R.string.verify_email_still_unverified)`
inside the `onSuccess = { verified -> ... }` fold branch — `it` there is the `Boolean` result, which
has no `getString`. Fixed by adding `val context = LocalContext.current` (matching `LoginScreen`'s
existing pattern) and calling `context.getString(...)`. Also the brief fully-qualified the email
icon as `androidx.compose.material.icons.Icons.Filled.Email`, which doesn't compile — `Icons.Filled.Email`
is a Kotlin extension property, not a nested member, so full qualification doesn't resolve it; fixed
by importing `androidx.compose.material.icons.Icons` + `androidx.compose.material.icons.filled.Email`
and using the plain reference (same style as `LoginScreen.kt`/`HelpAboutScreen.kt`, which already use
this icon). Caught both by the delegated `assembleDebug` build failing on the first pass; fixed here
and re-verified BUILD SUCCESSFUL.

**Localization:** 7 new string keys (`verify_email_title`, `verify_email_subtitle`,
`verify_email_continue`, `verify_email_still_unverified`, `verify_email_resend`,
