### Widget photos — REAL root cause found + fixed (verified on S8)

Sessions 1–2 misdiagnosed this as a Samsung One UI `file://` quirk. The actual bug: both widget
decoders (`OnThisDayWidgetProvider.decodeThumbnail`, `RecentEntriesWidgetService.decodeThumbnail`)
ran the `inJustDecodeBounds` bounds pass as:

```kotlin
WidgetPhotos.openStream(context, uri)?.use { BitmapFactory.decodeStream(it, null, boundsOpts) } ?: return null
```

`BitmapFactory.decodeStream` **returns null by design** when `inJustDecodeBounds = true` (it only
fills `boundsOpts`), so `?: return null` misread that expected-null as a failure and aborted the
function *before the real decode ever ran*. Confirmed on-device via temporary logcat: `streamNull=false`
(stream opened fine, cache file valid) but the `decode bounds=` log never printed → `bitmap=nullxnull`.

Fix: guard the **stream**, not the decode result —
`(WidgetPhotos.openStream(context, uri) ?: return null).use { BitmapFactory.decodeStream(it, null, boundsOpts) }`
then `if (boundsOpts.outWidth <= 0 || boundsOpts.outHeight <= 0) return null`. Applied to both decoders.
On-device (S8 SM-G955F): On This Day hero shows the Sagrada Família photo; all four Recent-entries rows
show real thumbnails. **General rule: never `?: return null` on the result of an `inJustDecodeBounds`
`decodeStream` — that result is legitimately null.**

The `WidgetPhotos.openStream` `file://`-via-`FileInputStream` split (from session 2) is kept — it's a
correct, separate fix — but was necessary-not-sufficient; the null-guard bug above was the real blocker.
